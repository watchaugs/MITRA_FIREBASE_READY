Before you begin

A short inventory. If anything below is missing, stop and arrange it first — the journey is much shorter once everything is on the desk.

One

A working laptop with internet — Windows, macOS, or Linux. All three work the same way.

Two

A Google account. Personal Gmail is fine. A Google Workspace email is better for an institution.

Three

A credit or debit card to enable Google Cloud billing. You get ₹25,000 of free credit at signup.

Four

The MITRA_FIREBASE_READY folder unzipped to your desktop. That's this code.

Five

About three quiet hours. Most of it is waiting — keep tea nearby.

Six

A notebook. You will write down a few names and IDs as you go.

01Tools

Install the two command-line tools

A one-time setup. Think of it as installing two specialised remote controls — one for Google Cloud, one for Firebase.

You will use two tools throughout this guide:

  • gcloud — Google Cloud's command-line tool. Used to create cloud resources.
  • firebase — Firebase's command-line tool. Used to publish the dashboard's web pages.

Both are free. Both run on your laptop and talk to Google's servers on your behalf. Pick your operating system below.

What the installers look like illustration
After installing each tool, you'll run a version check to confirm it works.

If you are on Windows

  1. Open your browser and go to cloud.google.com/sdk/docs/install. Click Windows installer, run the downloaded .exe, and accept all defaults.
  2. When the installer finishes it offers to run gcloud init — say yes. A browser window opens; sign in with the Google account you'll use for MITRA.
  3. Open PowerShell (Windows key, type "PowerShell", press Enter). Run:
powershell · verify gcloud
gcloud --version

You should see four or five lines listing the SDK version. If you see "command not found", close PowerShell and open a fresh window — the installer needs that.

Now install Node.js (Firebase CLI runs on it). Go to nodejs.org, download the LTS version, run the installer, accept defaults. Then back in PowerShell:

powershell · install firebase cli
npm install -g firebase-tools

If you are on macOS

Open Terminal (Spotlight: ⌘+Space, type "Terminal"). If you have Homebrew installed:

terminal · macOS with Homebrew
brew install --cask google-cloud-sdk
brew install node
npm install -g firebase-tools

If you don't have Homebrew, use the official installers at cloud.google.com/sdk/docs/install and nodejs.org instead.

If you are on Linux

terminal · linux (debian/ubuntu)
curl https://sdk.cloud.google.com | bash
exec -l $SHELL
sudo apt-get install -y nodejs npm
sudo npm install -g firebase-tools

Verify both tools work

any terminal · final check
gcloud --version
firebase --version
node --version
How you'll know it worked

Each command prints a version number. gcloud shows something like "Google Cloud SDK 480.x", firebase shows "13.x.x", and node shows "v20.x.x" or higher.

Common stumble

On Windows, if PowerShell says "command not found", you almost certainly need to close every PowerShell window and open a brand-new one. Installers update your PATH only for new windows.

02Project

Create your Google Cloud project

A project is the address book under which everything else lives — database, storage, server, secrets. Each project has a unique ID you will use again and again.

Open console.cloud.google.com in your browser. Sign in with the same Google account you used in step 1.

At the top of the page there is a project dropdown — it usually says "Select a project" or shows an existing project name. Click it. In the dialog that opens, click NEW PROJECT.

The New Project dialog illustration · console.cloud.google.com
The project ID is auto-generated. Write it down — you'll type it many times.
  1. For Project name, type MITRA Dashboard.
  2. Look at the auto-generated Project ID just below — it looks like mitra-dashboard-481204. Click the edit pencil if you want a tidier ID. Write this ID down in your notebook; you cannot change it later.
  3. Leave Organization as is (most personal accounts say "No organization" — that's fine).
  4. Click CREATE. A notification appears in the top-right when the project is ready, usually within 20 seconds.
  5. When the project finishes creating, click the notification, or use the project dropdown at the top to switch to it. The page header should now show MITRA Dashboard.

Now tell gcloud on your laptop which project to use. Open your terminal (PowerShell on Windows, Terminal on macOS/Linux) and run, replacing the ID with yours:

terminal · point gcloud at your project
gcloud config set project YOUR-PROJECT-ID
How you'll know it worked

Running gcloud config get-value project prints your project ID back. The console header shows "MITRA Dashboard" in the top-left dropdown.

03Billing

Turn billing on

Google offers ₹25,000 of free credit for the first 90 days, but they still need a card on file. Without billing, the database and server services refuse to start.

In the cloud console, open the navigation menu at the top-left (☰) and click Billing. If this is your first project, you'll be prompted to set up a billing account.

Billing account setup illustration · cloud console
Indian cards (Visa / Mastercard / RuPay) all work. UPI does not work for cloud billing.
  1. Choose your Country — India.
  2. Choose Account type — Individual (or Business if billing to your school).
  3. Enter your name, address, and card details. Google will charge ₹2 to verify the card and immediately refund it.
  4. Once submitted, you land on the Billing overview page. Click Link a billing account if prompted, and select your project MITRA Dashboard.
What this will actually cost

Once your free credit is used or expires, expect roughly ₹900 – ₹2,200 / month for a small school deployment: most of the cost is the Cloud SQL database (~₹700/mo for a db-f1-micro instance kept running). Cloud Run is free if no one is using the dashboard.

You can set a budget alert at Billing → Budgets & alerts → Create Budget. Set ₹2,500/month with an email notification at 50% and 90%. Highly recommended.

Don't skip this

If billing isn't enabled, step 5 will fail with messages about "billing account required". The error is honest. Come back here.

04Services

Enable the seven APIs you'll use

Google services are off by default. Turning them on is a two-minute task — one command does all seven.

Each cloud service (the database, the server, the storage, etc.) is gated behind an "API" that must be enabled in your project. Doing it from the command line is far faster than clicking through seven pages.

The seven APIs you're enabling visual reference
Each box is a separate service. You're authorising your project to use all seven.

Copy the command below and run it in your terminal. It enables all seven APIs at once. It takes about 60 seconds.

terminal · enable all seven
gcloud services enable \
  run.googleapis.com \
  sqladmin.googleapis.com \
  storage.googleapis.com \
  secretmanager.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  firebasehosting.googleapis.com

Windows PowerShell users: replace each backslash \ with a backtick `, or put it all on one line separated by spaces.

How you'll know it worked

The terminal prints "Operation finished successfully." Run gcloud services list --enabled to confirm — all seven should appear.

Common stumble

If you see "billing must be enabled", go back to step 3 — the project doesn't yet have a billing account linked.

05Database

Create the Cloud SQL Postgres instance

This is where every student record, every quiz, every analytics event will live. It takes about 8 minutes to create — start it and step away for tea.

You need to pick a few things:

  • Instance name — name of the database server. Use mitra-db.
  • Region — where the data lives geographically. For India use asia-south1 (Mumbai) or asia-south2 (Delhi).
  • Tier — server size. db-f1-micro is the smallest and cheapest; perfect for a single school. Upgrade later if needed.
  • Postgres password — for the built-in postgres superuser. Make it long. Save it in your password manager.
Creating mitra-db in Mumbai (asia-south1) illustration · cloud console
You can also do this from the console UI — same fields, same result. The CLI is faster.

Run this in your terminal. Replace CHOOSE-A-STRONG-PASSWORD with a long random password (24+ characters, mixed case, digits, symbols). Save it now — you cannot recover it later.

terminal · create mitra-db (takes ~8 min)
gcloud sql instances create mitra-db \
  --database-version=POSTGRES_15 \
  --tier=db-f1-micro \
  --region=asia-south1 \
  --root-password=CHOOSE-A-STRONG-PASSWORD \
  --storage-size=10 \
  --storage-auto-increase \
  --backup-start-time=02:00 \
  --availability-type=zonal

Once the instance is ready (you'll see "Created…"), create the actual database mitra inside it, and a dedicated application user:

terminal · create database and app user
gcloud sql databases create mitra --instance=mitra-db

gcloud sql users create mitra-app \
  --instance=mitra-db \
  --password=ANOTHER-STRONG-PASSWORD

Get the instance's connection name. You'll need it in step 8. Write it down.

terminal · get the connection name
gcloud sql instances describe mitra-db --format='value(connectionName)'

It will look like: mitra-dashboard-481204:asia-south1:mitra-db. That is the connection name. Copy it into your notebook.

How you'll know it worked

Run gcloud sql instances list. You should see mitra-db with status RUNNABLE.

Two passwords; two roles

The postgres user is the database root account — keep it for emergencies. The mitra-app user is what the server uses day-to-day. Never use postgres from the application.

06Storage

Create the Cloud Storage bucket

A bucket is a folder in the cloud. When teachers upload quiz spreadsheets, ad images, or Unity AR assets, they go here — not into the database.

Bucket names are globally unique across all of Google Cloud — no two customers can have the same one. So we'll prefix yours with your project ID. Run:

terminal · create bucket (replace project id)
gcloud storage buckets create gs://YOUR-PROJECT-ID-mitra-uploads \
  --location=asia-south1 \
  --uniform-bucket-level-access \
  --public-access-prevention

The flags ensure the bucket is locked down — files inside cannot be made public by accident. The server will hand out short-lived signed URLs when a teacher legitimately needs to download something.

The bucket structure MITRA will use visual reference
The server creates these folders on demand. You don't need to make them yourself.

Write down the full bucket name (including the gs:// prefix). You'll paste it as a secret in step 8.

How you'll know it worked

gcloud storage buckets list shows your new bucket. The console at Cloud Storage → Browser shows it as well.

07Secrets

Store your secrets safely

Passwords and signing keys never go in code, never in environment variables, never in screenshots. They go in Secret Manager — a vault Cloud Run can read from at startup.

You're going to store six secrets. Each one is created with a short command. Run them one at a time. Replace the example values with your own.

Generate two strong JWT signing keys first

JWT secrets sign the login tokens. They must be at least 32 bytes of random data. On macOS/Linux:

terminal · generate two random keys
openssl rand -base64 48
openssl rand -base64 48

On Windows PowerShell, use this instead:

powershell · generate two random keys
[Convert]::ToBase64String((1..48 | ForEach-Object {Get-Random -Maximum 256}))
[Convert]::ToBase64String((1..48 | ForEach-Object {Get-Random -Maximum 256}))

Copy each output — you'll paste them into the next two commands.

Create the six secrets

terminal · six secrets, one at a time
# 1. The app-user database password (from step 5)
echo -n 'ANOTHER-STRONG-PASSWORD' | gcloud secrets create mitra-db-password --data-file=-

# 2. JWT secret (paste your first openssl output)
echo -n 'YOUR-FIRST-RANDOM-KEY' | gcloud secrets create mitra-jwt-secret --data-file=-

# 3. JWT refresh secret (paste your second openssl output)
echo -n 'YOUR-SECOND-RANDOM-KEY' | gcloud secrets create mitra-jwt-refresh-secret --data-file=-

# 4. Storage bucket name (from step 6, without the gs:// prefix)
echo -n 'YOUR-PROJECT-ID-mitra-uploads' | gcloud secrets create mitra-storage-bucket --data-file=-

# 5. Cloud SQL connection name (from step 5)
echo -n 'YOUR-PROJECT:asia-south1:mitra-db' | gcloud secrets create mitra-sql-connection --data-file=-

# 6. Email password for password-reset emails (Gmail app password)
echo -n 'YOUR-GMAIL-APP-PASSWORD' | gcloud secrets create mitra-smtp-password --data-file=-
The vault, listed illustration · Secret Manager
Secrets are encrypted at rest and never logged.
About the Gmail app password

For password-reset emails, MITRA can send via your Gmail. In your Google account, go to Security → 2-Step Verification → App passwords, generate one for "Mail", and paste it as the sixth secret above. If you'd rather use a different SMTP provider (your school's, or SendGrid), the field names change — see .env.example.

Allow Cloud Run to read the vault

Find your project number (different from project ID):

terminal · get project number
gcloud projects describe $(gcloud config get-value project) --format='value(projectNumber)'

It's a long number like 182740321987. Use it below — this command grants the Cloud Run service permission to read your secrets at startup:

terminal · grant access to all six secrets
PROJECT_NUMBER=YOUR-PROJECT-NUMBER
for secret in mitra-db-password mitra-jwt-secret mitra-jwt-refresh-secret mitra-storage-bucket mitra-sql-connection mitra-smtp-password; do
  gcloud secrets add-iam-policy-binding $secret \
    --member=serviceAccount:${PROJECT_NUMBER}-compute@developer.gserviceaccount.com \
    --role=roles/secretmanager.secretAccessor
done
How you'll know it worked

gcloud secrets list shows all six secrets. Each add-iam-policy-binding prints "Updated IAM policy".

08Deploy

Deploy the API to Cloud Run

The biggest step. You're packaging the Node.js server into a container and shipping it to Google's edge. The included deploy script does the heavy lifting.

Open a terminal and navigate into the MITRA_FIREBASE_READY folder. For example:

terminal · go to the project folder
cd ~/Desktop/MITRA_FIREBASE_READY

(Adjust the path if you unzipped somewhere else. On Windows it might be cd C:\Users\YOU\Desktop\MITRA_FIREBASE_READY.)

The folder contains a script that builds the container, pushes it to Artifact Registry, and deploys it to Cloud Run — wired up to your Cloud SQL instance and your secrets. Run it:

On macOS / Linux

terminal · one-shot deploy
bash scripts/deploy.sh

On Windows (PowerShell)

powershell · one-shot deploy
powershell -ExecutionPolicy Bypass -File scripts\deploy.ps1
What you'll see during deploy terminal output
The Service URL at the end is what Firebase Hosting will forward API requests to.

While the build runs, the script will tell you exactly what's happening at each stage. Don't close the terminal until you see "Done".

How you'll know it worked

The script prints the final Service URL — something like https://mitra-api-xxxxxxxx-as.a.run.app. Copy this URL into your notebook. Open it in your browser and add /health to the end — you should see {"status":"ok"}.

If the deploy fails

The most common cause is a missed step. The script will say which one. The fixes are usually: re-run step 4 (APIs), re-run step 7 (grant secret access), or re-check the project ID with gcloud config get-value project. Re-run deploy.sh — it's safe to run twice.

09Schema

Run the database migrations

The database exists but it's empty. Migrations create every table — users, schools, advertisements, quizzes — in the correct order.

The MITRA folder contains a db/migrations/ directory with five versioned SQL files. They build the entire schema, in order, exactly once. A small Node script applies them and remembers which ones have run.

You'll connect to Cloud SQL from your laptop via the Cloud SQL Auth Proxy — Google's secure tunneling tool. Start it in its own terminal window:

terminal — window #1 · start the proxy (keep open)
gcloud sql connect mitra-db --user=mitra-app --database=mitra --quiet
# Or, if that doesn't work, use the standalone proxy:
# cloud-sql-proxy YOUR-PROJECT:asia-south1:mitra-db --port=5432

The simpler path on most setups: use the included migrate script which talks to Cloud SQL through the Cloud Run service itself. From the project folder:

terminal · run migrations (one command)
gcloud run jobs create mitra-migrate \
  --image=asia-south1-docker.pkg.dev/$(gcloud config get-value project)/mitra/mitra-api:latest \
  --region=asia-south1 \
  --command=node \
  --args=db/migrate.js \
  --set-cloudsql-instances=$(gcloud sql instances describe mitra-db --format='value(connectionName)') \
  --set-secrets=DB_PASSWORD=mitra-db-password:latest,JWT_SECRET=mitra-jwt-secret:latest,JWT_REFRESH_SECRET=mitra-jwt-refresh-secret:latest \
  --set-env-vars=DB_USER=mitra-app,DB_NAME=mitra,DB_HOST=/cloudsql/$(gcloud sql instances describe mitra-db --format='value(connectionName)')

gcloud run jobs execute mitra-migrate --region=asia-south1 --wait
The five migrations, in order visual reference
If you ever re-run migrations, only new ones are applied — safe to repeat.
How you'll know it worked

The last line of output reads ✓ Applied 5 migrations. Database is up to date. You can also confirm by visiting Cloud SQL → mitra-db → Databases → mitra → tables list in the console. You should see users, schools, advertisements, etc.

If you already had data in your old Postgres

The migration job creates an empty schema. To move your existing data over, follow MIGRATION_GUIDE.md next — it walks through pg_dump, the proxy, and psql restore. If MITRA is brand-new, skip that and continue to step 10.

10Admin

Seed the first admin user

The database has tables but no people in it yet. Create your first master admin so you can log in.

This is a one-time bootstrap. The seed script reads two environment variables — an email and a password — and creates a single master_admin account. After this, you add all other users from inside the dashboard.

Choose a strong password (at least 12 characters, mixed case, digits, symbols). The seed refuses to run on weak passwords.

terminal · seed the first admin
gcloud run jobs create mitra-seed \
  --image=asia-south1-docker.pkg.dev/$(gcloud config get-value project)/mitra/mitra-api:latest \
  --region=asia-south1 \
  --command=node \
  --args=db/seed.js \
  --set-cloudsql-instances=$(gcloud sql instances describe mitra-db --format='value(connectionName)') \
  --set-secrets=DB_PASSWORD=mitra-db-password:latest \
  --set-env-vars=DB_USER=mitra-app,DB_NAME=mitra,DB_HOST=/cloudsql/$(gcloud sql instances describe mitra-db --format='value(connectionName)'),SEED_ADMIN_EMAIL=you@yourschool.edu.in,SEED_ADMIN_PASSWORD=PickAStrongPassword!2026

gcloud run jobs execute mitra-seed --region=asia-south1 --wait
The master_admin you're creating illustration
Once seeded, you can delete the SEED_ADMIN_PASSWORD from the Cloud Run Job's environment.
How you'll know it worked

The job output reads ✓ Created master_admin: you@yourschool.edu.in. The job exits with status zero.

Important — delete the seed credentials after

Your admin password is currently visible in the Cloud Run Job definition. After seeding succeeds, clear it:

terminal · clear the seed password
gcloud run jobs delete mitra-seed --region=asia-south1 --quiet
11Hosting

Publish the dashboard to Firebase Hosting

Firebase Hosting serves the dashboard's HTML, CSS, and JavaScript globally and securely, with HTTPS by default. It also forwards all /api/ calls to the Cloud Run service you deployed.

First, log into Firebase from your terminal:

terminal · login to firebase
firebase login

A browser opens; sign in with the same Google account.

Now tell Firebase which project to use. From inside the MITRA_FIREBASE_READY folder:

terminal · point firebase at your project
firebase use --add

It lists your Google Cloud projects. Choose the MITRA one. When asked for an alias, type default.

The folder already contains a firebase.json configured to serve the public/ directory and forward /api/** to the Cloud Run service. Deploy it:

terminal · deploy hosting
firebase deploy --only hosting
How requests flow once published architecture
Firebase Hosting handles static files; /api/** proxies to Cloud Run.
How you'll know it worked

The terminal prints a Hosting URL like https://your-project.web.app (and your-project.firebaseapp.com). Open it in your browser — you should see the MITRA login page.

Custom domain (optional, can do later)

If you own a domain like mitra.yourschool.edu.in, you can connect it from Firebase Console → Hosting → "Add custom domain". Firebase auto-provisions an SSL certificate. The DNS changes propagate in a few hours.

12Login

Log in for the first time

The proof. You'll sign in as the admin you seeded in step 10, change your password, and create one more test user.

Open your Hosting URL from step 11 in a private/incognito window (to avoid stale caches).

The first-login flow illustration · dashboard
Treat your master admin password like a master key. Rotate it now.
  1. Enter your seed email and password.
  2. Click Sign in. You should land on the dashboard's welcome screen.
  3. Immediately go to Settings → Profile → Change password and set a new password (different from the seed one).
  4. Go to Users → Add user. Create a test user with role school_admin. The system sends them a one-time password-reset link (via the SMTP password you set in step 7).
How you'll know it worked

You can sign in, change your password, sign out, and sign back in with the new one. The added test user receives an email with a reset link.

If the login spins or errors

Open the browser's DevTools (right-click → Inspect → Network tab). If you see /api/auth/login returning 502 or 503, the Cloud Run service is cold-starting; wait 15 seconds and retry. If it returns 401, the password is wrong — go back to step 10 and re-seed. If it returns 500, look at the Cloud Run logs: gcloud run services logs read mitra-api --region=asia-south1 --limit=50.

13End-to-end

Verify uploads and downloads

The last loop. Upload an image as an advertisement, see it land in Cloud Storage, then download it back through the dashboard.

This is the only step that exercises every piece at once: the browser, Firebase Hosting, the Cloud Run API, Cloud SQL (records), Cloud Storage (the file), and the signed-URL flow.

  1. In the dashboard, go to Advertisements → New advertisement.
  2. Fill the title, set status to draft, and upload a small JPG or PNG (under 5 MB).
  3. Click Save. The file uploads, the row is created, you return to the list.
  4. Click on the new ad. Click Download media. The browser should fetch the file via a short-lived signed URL.
  5. Open Cloud Storage Browser. Navigate into your bucket → ad_media/. The uploaded file is there.
Upload → store → signed-URL → download end-to-end
Signed URLs expire after 15 minutes — fresh ones are minted on demand.
How you'll know it worked

(1) The upload progresses to 100%. (2) The new ad appears in the list. (3) The download fetches the file (note the URL contains storage.googleapis.com with a signature). (4) The Cloud Storage Browser shows the file with the same size as your original.

If upload fails

Look at the response in DevTools → Network. Common causes: file too large (default cap 50 MB; raise MAX_UPLOAD_SIZE if needed), wrong MIME type (only specific image / video / xlsx types are allowed per category), or the service account is missing storage permissions (re-run the IAM binding from step 7).

Fin

You're live.

The dashboard your teachers and admins will use every day is now on Google's infrastructure, protected by HTTPS, backed up nightly, and ready for production.

What next

A short list. None are urgent, but doing them within the first week sets you up well:

  • Budget alert. Billing → Budgets & alerts → Create, set to ₹2,500/month with 50% and 90% email triggers.
  • Custom domain. Connect mitra.yourschool.edu.in via Firebase Console → Hosting.
  • Test backups. Cloud SQL → mitra-db → Backups. Confirm the nightly backup ran. Once a quarter, restore a backup to a temporary instance to verify the procedure.
  • Read the security audit. Open SECURITY_AUDIT.md. It lists every issue found in the original code, what was fixed, and a short list of things still on your shoulders (rotating JWT keys quarterly, reviewing user list quarterly).
  • Plan the AR app. The dashboard you just deployed serves the AR app's API too. When the Unity smartphone app is ready, it points to the same Cloud Run URL.

Welcome to operating a real cloud service. Take care of it the way you'd take care of a small garden — quiet, regular attention beats heroic rescues every time.
